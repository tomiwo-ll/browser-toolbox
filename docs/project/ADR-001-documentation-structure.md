# ADR-001: ドキュメント配置とADR運用を定義する

## ステータス

Accepted

## 日付

2026-05-06

## 背景

このリポジトリでは、ブラウザ上で処理が完結する複数のツールを継続的に追加する。リポジトリ全体の方針、共通仕様、ツール単体の仕様が混ざると、要件や意思決定の追跡が難しくなる。

また、機能拡張や仕様変更では、過去の判断を上書きせずに変更理由を追える必要がある。

## 決定

- リポジトリ全体、共通仕様、共通運用についてのドキュメントは `docs/project/` に置く。
- 特定ツール単体についてのドキュメントは `docs/tools/<tool-en-name>/` に置く。
- 各ディレクトリには、必要に応じて要件定義書、仕様書、トレーサビリティ確認表、ADR、その他補助資料を置く。
- 要件定義書、仕様書、トレーサビリティ確認表、ADRは `docs/project/templates/` のテンプレートを使って作成する。
- なんらかの意思決定を行った場合はADRを作成する。機能拡張や仕様変更では、基本的にADRを先に作成し、それをもとに既存の要件定義書、仕様書、トレーサビリティ確認表を更新する。
- ADRファイル名は `ADR-XXX-<title>.md` とする。`XXX` はディレクトリごとの三桁ゼロ埋め連番、`<title>` は kebab-case とする。
- 既存ADRは原則として編集しない。新しいADRを作成し、既存ADRとの関係を新ADR側に記載する。
- 既存ADRが陳腐化する場合のみ、既存ADRに最小限の状態変更または参照追記を行い、どのADRに置き換えられたか分かる状態にする。

## 採用しなかった案

- `docs/features/<yyyy-mm-dd>-<feature-slug>/` に機能単位で置く案。
  - 日付ベースのため、リポジトリ共通仕様とツール単体仕様の境界が分かりづらい。
- すべてのドキュメントを `docs/project/` に集約する案。
  - ツール数が増えたときに、ツール単体の仕様と共通仕様が混ざりやすい。
- 既存ADRを直接更新し続ける案。
  - 判断の履歴が失われやすい。

## 影響

- `AGENTS.md` のドキュメント作成手順を更新する。
- `docs/project/overview.md` にドキュメント構造を追記する。
- `docs/project/templates/` に要件定義書、仕様書、トレーサビリティ確認表、ADRのテンプレートを追加する。
- 今後のツール追加では `docs/tools/<tool-en-name>/` を使う。

## 関連ADR

- なし。
