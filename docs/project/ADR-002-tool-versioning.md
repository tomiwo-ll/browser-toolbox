# ADR-002: ツール単位のバージョニングを定義する

## ステータス

Accepted

## 日付

2026-05-06

## 背景

このリポジトリでは、複数のブラウザ完結ツールを同じサイト上で提供する。各ツールは追加、拡張、修正のタイミングが異なり、あるツールの変更が別ツールの互換性や成熟度を意味するとは限らない。

最終的には、各ツールのページ上で現在バージョンとバージョン更新履歴を表示したい。そのため、表示に使うデータと、仕様・意思決定の記録をどこで管理するかを決める必要がある。

## 決定

- バージョニングはツール単位で行う。
- 各ツールは Semantic Versioning を使い、`MAJOR.MINOR.PATCH` 形式で管理する。
- `MAJOR` は既存の入力、出力、操作、期待される成果物との互換性を壊す変更に使う。
- `MINOR` は後方互換の機能追加、出力形式追加、設定追加、表示可能な履歴項目追加に使う。
- `PATCH` はバグ修正、表示修正、性能改善、仕様に影響しない内部改善に使う。
- 現在バージョンとページ表示用の更新履歴は、ツールの実装メタデータを正本とする。
- ツール実装メタデータは `src/tools/<tool-en-name>/metadata.ts` を基本配置とする。ツール構成が導入されるまでは、既存のツール定義に同等の情報を置いてよい。
- 仕様・判断履歴・レビュー用の変更履歴は `docs/tools/<tool-en-name>/` に置く。
- 仕様変更や機能拡張を行う場合はADRを作成し、必要に応じて要件定義書、仕様書、トレーサビリティ確認表、更新履歴を更新する。

## 採用しなかった案

| 案 | 採用しなかった理由 |
| --- | --- |
| リポジトリ全体で単一バージョンを持つ | ツールごとの成熟度や互換性を表現できない。 |
| ドキュメントの `changelog.md` をページ表示の正本にする | 実装で表示するデータと文書が二重管理になりやすい。 |
| Git tag だけでツールバージョンを表現する | 複数ツールの個別リリース履歴を扱いづらい。 |

## 影響

- ツールページでは、実装メタデータから現在バージョンと更新履歴を表示する。
- ツール単体の仕様書テンプレートには、現在バージョンと更新履歴表示の項目を含める。
- トレーサビリティ確認表では、仕様と実装メタデータのバージョン・更新履歴が一致していることを確認する。
- `AGENTS.md` と `docs/project/overview.md` にツール単位のバージョニング方針を記載する。
